# AGENTS.md

Документ фиксирует рабочие договоренности для всех ассистентов/чатов в этом проекте.

## 1. Режим работы

1. Двигаться очень маленькими шагами.
2. Делать только то, что явно запросил пользователь в текущем шаге.
3. Не добавлять лишнюю верстку, стили или архитектуру без отдельного запроса.
4. Перед изменениями сначала коротко подтвердить, что будет сделано.

## 2. Коммуникация

1. Если задача неоднозначна, сначала задать уточняющие вопросы.
2. Если есть несколько вариантов, сначала предложить принятые практики, потом править код.
3. Не делать крупные рефакторы без явного согласия.
4. Отвечать по делу, без лишнего текста.
5. При неоднозначности задавать один короткий уточняющий вопрос перед правками, если ответ влияет на поведение.

## 3. Изменения в коде

1. Сохранять существующую структуру проекта, если не просили ее менять.
2. В Next.js:
   - файл маршрута: `page.tsx`
   - имя компонента внутри файла: осмысленное (`HomePage`, `CatalogPage` и т.д.), не обезличенное.
3. Не менять стиль кода/форматирование отдельными массовыми правками без запроса.
4. Не добавлять новые зависимости без необходимости и явного уведомления.
5. Переменные деструктурировать на максимально верхнем уровне.
6. Строковые константы и одиночные символы выносить в константы наверху компонента.
7. Такие константы именовать в `UPPER_SNAKE_CASE`.
8. Короткие коллбеки писать в сокращенной форме: с деструктуризацией прямо в параметрах и без лишнего блока `return`, если логика помещается в одно выражение.
9. Повторяющиеся значения в SCSS выносить в переменные наверху файла.
10. Для условного рендера в JSX использовать `&&` там, где не нужен альтернативный блок.
11. В SCSS использовать вложенную структуру через родительский блок и `&__...`, если стили относятся к одному компоненту.
12. Файлы `.md`, `.ts`, `.tsx`, `.scss` сохранять в `UTF-8` (без BOM).

## 4. Безопасность изменений

1. Не трогать несвязанные файлы.
2. Не откатывать пользовательские изменения без явной просьбы.
3. Если во время работы обнаружены неожиданные изменения, остановиться и уточнить у пользователя.

## 5. Проверка результата

1. После правок сообщать, что именно изменено (коротко, по файлам).
2. Если проверка не выполнена, явно писать об этом.
3. Для каждой задачи по возможности давать следующий минимальный шаг.
4. Минимум перед сдачей: по возможности запускать релевантную проверку (`pnpm lint`, `pnpm test` или точечная проверка затронутого модуля).
5. Задача считается завершенной, если выполнен явный запрос пользователя и нет побочных правок вне области задачи.
